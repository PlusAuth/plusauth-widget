</JsonKey>


You can also use PlusAuth widget as a npm library. To do so install it to your project with:

```shell
# with npm
npm install @plusauth/plusauth-widget

# with yarn
yarn add @plusauth/plusauth-widget

# with pnpm
pnpm add @plusauth/plusauth-widget
```

# Usage

## Constructor

`PlusAuthWidget` constructor signature:

```ts
new PlusAuthWidget(
  container?: Element | string,
  settings?: Partial<IWidgetSettings>,
  context: Partial<IPlusAuthContext>
)
```

`container` can be a selector string (for example `'#widget'`) or an element reference.

`settings` controls mode, fields, theme, locale, and behavior.

`context` is the PlusAuth runtime payload (`window.PlusAuth` in hosted pages).

## Basic Initialization

Initialize with explicit target, settings, and context:

```html
<script >
  new PlusAuthWidget('#widget', {
    mode: 'login'
  }, window.PlusAuth)
</script>
```

or

```js
import PlusAuthWidget from '@plusauth/plusauth-widget'

new PlusAuthWidget('#widget', {
  mode: 'login'
}, window.PlusAuth)
```

If you don't have context yet in development, pass an empty object:

```javascript
new PlusAuthWidget('#widget', { mode: 'login' }, {})
```

## API Base URL

Set `apiUrl` when widget requests should be sent to a specific origin instead of the current page origin.

```javascript
new PlusAuthWidget('#widget', {
  mode: 'login',
  apiUrl: 'https://tenant.example.com'
}, window.PlusAuth)
```

Internal form submissions use this base URL together with the current page path.

## Runtime API

The widget instance exposes a few helpers that are useful after initialization:

- `widget.view`: reactive settings object. You can switch `mode` at runtime and update runtime-facing settings.
- `widget.i18n`: translation helper. Use `widget.i18n.t(...)` for lookups and `widget.i18n.locale = 'tr'` to switch locale.
- `widget.http`: fetch wrapper used by internal flows. It uses `apiUrl` if provided.

```javascript
const widget = new PlusAuthWidget('#widget', {
  mode: 'login',
  apiUrl: 'https://tenant.example.com'
}, window.PlusAuth)

widget.view.mode = 'mfa-otp'
widget.i18n.locale = 'en'
widget.http.post('/custom-endpoint', { body: { hello: 'world' } })
```

## Mode Selection

Set an initial mode in constructor settings:

```javascript
const widget = new PlusAuthWidget('#widget', {
  mode: 'login'
}, window.PlusAuth)
```

Or switch mode at runtime:

```javascript
widget.view.mode = 'mfa-sms'
```

If `mode` is omitted, widget also resolves view from current URL path conventions.

## Mode Options Mapping

For field/action customization, `modeOptions` keys may differ from the mode string.

- `login` -> `login`
- `register` -> `signup`
- `password-recovery` -> `recovery`
- `reset-password` -> `resetPassword`
- `fill-missing` -> `fillMissing`
- `verify-email` -> `verifyEmail`
- `password-challenge` -> `passwordChallenge`
- `mfa-email` -> `emailMfa`
- `mfa-sms` -> `smsMfa`
- `mfa-otp` -> `otpMfa`
- `mfa-push` -> `pushMfa`
- `mfa-webauthn` -> `webauthnMfa`
- `mfa-fv` -> `fvMfa`
- `passwordless-email` -> `passwordlessEmail`
- `passwordless-sms` -> `passwordlessSms`
- `passwordless-otp` -> `passwordlessOtp`
- `passwordless-push` -> `passwordlessPush`
- `passwordless-webauthn` -> `passwordlessWebauthn`
- `account-linking` -> `accountLinking`

## Global Behavior Settings

You can configure social button appearance and footer behavior globally:

```javascript
new PlusAuthWidget('#widget', {
  socialLogin: {
    buttonVariant: 'circle' // or 'block'
  },
  footer: {
    enabled: false
  }
}, window.PlusAuth)
```

- `socialLogin.buttonVariant`: renders social buttons as `block` or `circle`
- `footer.enabled`: hides or shows the footer locale/legal links area

## Development And Testing

Useful local commands:

```shell
npm run storybook
npm run build-storybook
npm test -- --run
npm run test:storybook -- --run
npm run type-check
```

- `npm test -- --run`: runs unit tests for component specs under `tests/components`
- `npm run test:storybook -- --run`: runs Storybook interaction tests
- `npm run type-check`: validates the Vue and TypeScript surface used by the widget library

For customization work, Storybook interaction tests are usually the best first check because they exercise the rendered widget behavior rather than only static snapshots.
